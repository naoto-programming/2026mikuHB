# ビートソード 既存プロトタイプ修正設計

日付: 2026-07-12
対象ファイル: `index.html`（単一ファイル実装、約2500行）

## 背景

`index.html` には既に単人プレイの完成度が高いプロトタイプが実装済み（Web Audio APIによるビート駆動のリズムシステム、8キャラクター、能力・カウンター・成長システム、5種類の敵）。ただし以下が仕様書とズレている、またはコードとして不自然:

- ネットワーク層は未実装（`hostIP`表示等はダミー、WebSocket/WebRTC等のコードは存在しない）— 今回のスコープ外
- キャラクター数が8のままで、精霊騎士・竜騎士を削除したいという要望がある
- プロジェクトフォルダの画像アセットが一切使われておらず、全てCanvasベクター図形で代用されている
- 固有能力が全キャラ共通の「範囲内の敵へ固定ダメージ」処理になっており、仕様書が個別に説明する能力性能（連射・神速・百裂拳等）と一致しない
- カウンター成功時、実際には敵の攻撃が完全に無効化されず、スタン明けに同じ攻撃が着弾してダメージを受けてしまうバグがある

プロジェクトフォルダには追加で以下の素材が用意されている:

- BGM音源4種（ファイル名がBPMを表す）: `79拍:分1.mp3`(BPM79, 約47秒) `79拍:分2.mp3`(BPM79, 約126秒) `90拍:分.mp3`(BPM90, 約90秒) `110拍:分.mp3`(BPM110, 約111秒)
- 武器画像: `弓.png`（弓士の持ち手用）, `矢.png`（弓士の攻撃・連射能力の飛び道具用）

## スコープ

今回のスコープは、上記プロトタイプに対する **修正・機能追加** であり、LANマルチプレイの実通信実装は含まない。

## 0. BGM実音源への差し替え

現状 `AudioSystem` はキック/スネア/ハイハットをWeb Audio APIで毎拍合成しているのみで、実音源ファイルを再生していない。以下のように変更する。

- ステージ開始時、4種のBGMから**完全ランダムに1つ選択**する（ステージ番号による割当は行わない。BPMによる難易度上昇の仕組みは、この変更に伴い廃止し、代わりに敵構成・攻撃パターンの複雑化（既存の`stage`依存ロジック）のみで難易度を上げる）
- 選択したファイル名からBPM値を取得（`79拍:分1.mp3`→79、`79拍:分2.mp3`→79、`90拍:分.mp3`→90、`110拍:分.mp3`→110）し、既存の拍クロック計算（`getCurrentBeat`等、経過時間÷(60/BPM)）はそのまま流用する
- 音源は`decodeAudioData`で事前デコードし`AudioBufferSourceNode`で`ctx.currentTime`基準の`startTime`から再生開始する。この`startTime`を拍クロックの基準時刻と共通化し、音楽とノーツ判定のズレを防ぐ
- 曲は`loop = true`で再生し、ステージがちょうど1周で終わらない場合の保険とする（下記「1周年ステージ長」参照）
- 判定SE（Perfect/Great/Good/Miss、剣攻撃音、能力音、カウンター音）は既存の合成音のまま変更しない。キック/スネア/ハイハットの毎拍合成コードは削除する

### ステージ長を曲の長さに合わせる

現状 `StageManager.maxDistance` は固定値`STAGE_LENGTH=4000`で、実測で約33秒相当しかなく仕様書の「1ステージ約5分」からも実音源の尺（47〜126秒）からも乖離している。ステージ開始時に選択されたBGMの再生時間から `maxDistance = trackDurationSeconds * scrollSpeed * 60` を算出し、そのステージの目標距離として設定する。これにより「曲を1回再生し切る頃にステージが終わる」形になり、既存の距離ベースの敵出現・エリート出現・クリア判定ロジックはそのまま流用できる。

## 1. キャラクター削除

`CHARACTERS` 配列から `id: 'spirit'`（精霊騎士）と `id: 'dragon'`（竜騎士）のエントリを削除する。残り6キャラ（剣士1・弓士2・盗賊3・拳士4・獣人5・魔法使い6）は既に難易度順に連番のため、番号の振り直しは不要。

キャラクター選択画面（`setupUI`）・難易度表示（`★`/`☆`の8段階表示）も6段階に合わせて調整する。

## 2. 画像アセットの導入

現状 `Renderer.renderPlayer` / `renderEnemy` / `renderBackground` は全てCanvas図形描画。以下のように `Image` オブジェクトをプリロードし `drawImage` へ置き換える。

### キャラクター画像
`剣士.png` `弓士.png` `盗賊.png` `拳士.png` `獣人.png` `魔法使い.png`（全て279〜458px幅 × 550〜829px高の縦長全身絵）を各キャラの `char.id` に対応付け、`facing`が-1のときは `ctx.scale(-1,1)` で左右反転して描画する。攻撃・カウンター・能力発動時の既存のエフェクト（発光、パーティクル等）は画像の上から重ねて維持する。

魔法使いは `魔法使いの杖.png`（1040×1080、手持ちアクセサリ）、弓士は `弓.png`（手持ちアクセサリ）をそれぞれキャラ画像に重ねて描画する。弓士の通常攻撃・「連射」能力発動時には `矢.png` を飛び道具スプライトとして敵に向けて飛ばす演出を追加する。

### 敵画像
既存の `ENEMY_TYPES` に対して:
- `normal` → `通常敵.png`
- `flying` → `飛行敵.png`
- `ranged` → `弓敵.png`
- `defense` → `盾敵.png`
- `large` → `大型敵.png`
- `elite` → `大型敵.png` を流用。ただし `ctx.filter` またはオフスクリーンキャンバスで赤みを強めたティント + 発光(`shadowBlur`)を強化し、大型敵と視覚的に区別する

### 背景画像
`地面と空.png`（1452×448、横長のパララックス背景）をスクロール位置に応じて横方向にタイル描画し、既存の山・建物のベクター背景と差し替える。`太陽.png` を遠景の固定位置に1つ配置。`木.png` を中景レイヤーに一定間隔で反復配置し、既存のパララックス建物・柱の一部と置き換える。

### 武器アイコン
`剣.png`（172×172）はリズムUIのソードノーツ（現在 "Z" の文字を描いている丸アイコン）に重ねて描画し、視認性を上げる。

## 3. 敵タイプ追加

`ENEMY_TYPES` に2種追加する。

### 自爆敵（`suicide`）
- 画像: `自爆敵.png`
- 高速接近・低HP・接近すると短い予兆(`attackWarning`)を出し、予兆終了時に**命中の有無に関わらず自爆して消滅**（`dead = true`）し、範囲内のプレイヤーに大ダメージを与える
- カウンター成功時は爆発前に撃破され、ダメージを与えずに消滅する（既存のカウンター処理をそのまま利用できる）

### ヒーラー敵（`healer`）
- 画像: `ヒーラー敵.png`
- 遠距離維持型（プレイヤーに近づかず後方を維持）。攻撃タイマーが切れる代わりに、周期的に「自身を除く生存中の敵のうちHP割合が最も低い1体」を回復させる（回復量は固定値、既存の`heal`相当のロジックを敵側にも実装）
- 回復モーションもプレイヤーの攻撃と同様に`attackWarning`を出し、カウンターで中断・大ダメージを与えられる（プレイヤーへの直接ダメージは発生しない）

`StageManager.spawnEnemy` の出現テーブルにステージ進行に応じて追加する（自爆敵はstage2以降、ヒーラー敵はstage3以降を想定）。

## 4. 固有能力の個別実装

現状 `Game.update()` 内の能力完了処理は、キャラクターによらず「命中率(ratio)に応じた固定ダメージを画面内の敵全体に与える」処理のみ。仕様書の記述に沿って `char.id` ごとに分岐させる。

| キャラ | 能力名 | 新しい挙動 |
|---|---|---|
| 剣士 | 斬鉄剣 | 現行同様：前方広範囲の敵全体に一撃（据え置き） |
| 弓士 | 連射 | 画面内の敵（最大5体程度）に個別の飛び道具ヒットを発生させる多段ヒット処理 |
| 盗賊 | 神速 | ダメージ処理の代わりに、一定時間（例:5秒）ノーツ生成間隔を短縮／判定ウィンドウを有利にする速度バフを付与 |
| 拳士 | 百裂拳 | 至近距離の敵（1〜2体）に短時間で複数回連続ヒット |
| 獣人 | 獣王撃 | 画面内で最もHPが高い（またはエリート優先の）敵1体に対する単体超高威力ヒット |
| 魔法使い | メテオ | 現行同様の全体範囲ダメージだが、最大範囲・最大ダメージ係数に強化 |

全成功／一部成功／全Missによる性能スケーリング（現行の`hitCount/total`比率をダメージ・効果量に反映する仕組み）は共通の考え方として維持し、各キャラの新ロジックにも同様に適用する。

## 5. バグ修正：カウンター成功時に攻撃が無効化されない

現状、`Game.handleCounter()` と `Enemy.executeAttack()` が独立に判定を行っており、カウンター成功時は `Game.handleCounter()` 側で即座にダメージ・スタンを与え `counterable = false` にするが、`isAttacking` / `attackWarning` はリセットされない。そのためスタン(`stunTimer`)が明けた後、`Enemy.update()` の通常の攻撃タイマー処理が再開し、同じ攻撃が結局プレイヤーに命中してダメージを与えてしまう。

修正方針: `Game.handleCounter()` でカウンターが成立した敵に対し、`counterable = false` に加えて `isAttacking = false`, `attackWarning = false` も設定し、その攻撃サイクルを完全に終了させる。スタン明け後は `attackCooldown` から新しい攻撃サイクルとして再開する。

## 今回のスコープ外（変更しない）

- 接触ダメージ（`Game.update()`内の「敵に触れると自動でダメージ」処理）は現状維持
- キャラクターごとの操作難易度差（ノーツ密度・判定ウィンドウの違い）は仕様書に記載があるが未実装のまま。将来の拡張候補として記録するのみ
- LANマルチプレイの実通信（WebSocket/WebRTC等）、8人対応、専用サーバーなしのホスト方式は未実装のまま
