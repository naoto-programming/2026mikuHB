# ビートソード 戦闘演出・判定ポリッシュ設計

日付: 2026-07-12
対象ファイル: `game.js`, `index.html`, `style.css`

## 背景

`docs/superpowers/specs/2026-07-12-gameplay-overhaul-design.md`（ウェーブ制アリーナ化・カウンター全廃等、Task 1〜7実装済み）を実際にブラウザで確認したユーザーから、以下のフィードバックを得た。

- 剣攻撃の判定が向いている方向にしか出ない（ウェーブが左右両方から来るのに、片側にしか攻撃できない）
- 固有能力の演出が全キャラ共通で個性が感じられない
- 通常攻撃(Z)の演出が全キャラ共通の「剣の斬撃弧」になっているが、実際に剣を持つのは剣士だけで、他5キャラ（弓士=弓、盗賊=ナイフ、拳士=拳、獣人=爪、魔法使い=杖）は得物が違う
- 飛行敵に剣攻撃が全く当たらない
- リズムノーツ（円形アイコン）が他のUI要素に隠れて見えにくい

## スコープ

先行実装（ウェーブ制・Zトリガー式ノーツバースト・カウンター全廃・ノックバック演出・UI重なり修正・6キャラクター・BGM実音源・画像アセット・自爆敵/ヒーラー敵）はすべて維持する。今回は戦闘判定・演出まわりのポリッシュに限定する。

## 1. 剣攻撃の判定を左右両方に広げる

### 現状
`Player.getAttackHitbox()` は `this.facing`（向いている方向）に応じて片側にしか判定ボックスを出さない。

```js
getAttackHitbox() {
    const range = this.getAttackRange();
    return {
        x: this.facing > 0 ? this.x : this.x - range,
        y: this.y - 50,
        w: range,
        h: 50,
    };
}
```

### 変更後
自分を中心に左右へ`range`ずつ広がる、向きに依存しない判定にする。

```js
getAttackHitbox() {
    const range = this.getAttackRange();
    return {
        x: this.x - range,
        y: this.y - 50,
        w: range * 2,
        h: 50,
    };
}
```

`facing`は見た目（スプライトの反転・演出の左右）にのみ使用し続け、当たり判定には影響しなくなる。

## 2. リズムノーツがUIに隠れる問題の修正

### 原因
`#gameContainer`内で`<canvas>`は`position`指定なしの通常フロー要素であり、CSSの仕様上、`position: absolute`を持つ兄弟要素（`#bottomHud`など、z-indexの有無に関わらず）は必ずCanvasより手前に描画される。`renderRhythmUI`はCanvas上でノーツを最前面に描画しているつもりでも、DOM側の`#beatBar`（黒背景+`.beat-target`のターゲット十字線）が上に重なって隠してしまう。

`#beatBar`はCanvas側の`renderRhythmUI`が既に同等の背景・ターゲットライン・ターゲットグローを描画しており、完全に重複した表示である。

### 変更後
DOM側の`#beatBar`要素（`.beat-target`を含む）を`index.html`/`style.css`/`game.js`（`#bottomHud`内の参照）から削除し、Canvas描画の`renderRhythmUI`のみに一本化する。これによりノーツを隠すものが無くなり、常に最前面に表示される。

`#bottomHud`は残りの`comboDisplay`・`tutorialBox`のみを含む形になる。

## 3. 通常攻撃(Z)の演出を6キャラ全員個別化する

### 現状
`Renderer.renderPlayer`の`if (p.isAttacking)`ブロックは、全キャラ共通のオレンジ色の斬撃弧を描画する。

### 変更後
`p.charId`に応じて以下の演出に分岐する（すべてCanvas描画、新規画像アセットは使わない）。

| キャラ | 演出 |
|---|---|
| 剣士 | 既存の斬撃弧（据え置き） |
| 弓士 | 手元から矢（既存のIMAGE_MANIFEST.weapons.arrow = 矢.png）を発射方向へ短距離飛ばす演出 |
| 盗賊 | 素早い二重のナイフスラッシュ（細い線を2本、X字状にクロスさせ素早くフェード） |
| 拳士 | 拳の位置から広がる衝撃波リング（円形、内側から外側へ拡大） |
| 獣人 | 平行な3本のクロー（爪）スラッシュ線 |
| 魔法使い | 杖の位置から放たれる魔法の煌めき（パーティクルバースト、紫〜青系） |

## 4. 固有能力(X)の演出を6キャラ全員個別化する

### 現状
`Renderer.renderPlayer`の`if (p.isUsingAbility)`ブロックは、全キャラ共通の青いオーラ＋回転パーティクルを描画する。

### 変更後
`p.charId`に応じて、実際の`applyAbility`の効果内容と対応する見た目に分岐する（色は`p.char.color`を基調にする）。

| キャラ | 能力 | 演出 |
|---|---|---|
| 剣士 | 斬鉄剣（全体範囲） | 横に広がる衝撃波（画面全体に広がるリング） |
| 弓士 | 連射（複数対象） | 複数方向へ飛ぶ矢のストリーク演出 |
| 盗賊 | 神速（速度バフ、ダメージなし） | 高速の残像・スピードライン演出 |
| 拳士 | 百裂拳（単体多段） | 連続する同心円の波紋 |
| 獣人 | 獣王撃（単体超高威力） | 一つの巨大な衝撃波リング |
| 魔法使い | メテオ（全体範囲） | 頭上から降ってくる隕石のようなグロー演出 |

## 5. 飛行敵にも剣攻撃を少し当てられるようにする

### 現状
`Player.getAttackHitbox()`の`h`（高さ）が50で、通常の地上敵にしか届かない。飛行敵は`GROUND_Y - 80 〜 GROUND_Y - 120`の高度に出現するため、現状のプレイヤー攻撃判定（`y-50 〜 y`）とは重ならず、剣で一切当たらない。

### 変更後
`Player.getAttackHitbox()`の`h`を50→110に広げ、判定の上端を引き上げる。

```js
getAttackHitbox() {
    const range = this.getAttackRange();
    return {
        x: this.x - range,
        y: this.y - 110,
        w: range * 2,
        h: 110,
    };
}
```

（項目1の左右判定拡張と合わせて適用する。`y`/`h`の変更により、飛行敵の出現高度帯のうち低めの範囲には当たるが、高く飛んでいる場合は引き続き当たりにくい「少し当たる」挙動になる）

## 6. 目の描画を削除する

### 現状
`Renderer.renderEnemy`では、目の描画（白目+黒目）が`if (sprite) {...} else {...}`の**外側**に置かれており、実画像スプライトを描画していても常に上から点目が重なって表示されてしまっている。`Renderer.renderPlayer`では目の描画はフォールバック（画像未読込時）分岐の中にあるため実害はないが、今回まとめて削除する。

### 変更後
`renderEnemy`の「// Eyes」ブロック（白目・黒目の`ctx.arc`呼び出し4回分）を削除する。`renderPlayer`のフォールバック分岐内の「// Eyes」ブロックも削除する（実画像には既に表情が描かれているため）。

## 7. 攻撃・能力演出をさらに豪華にする

項目3・4（キャラ別演出個別化）の各演出について、パーティクル数・グロー半径・残像の量を増やし、画面内で目立つ派手な見た目にする。具体的な倍率・パーティクル数は実装時に決める。

## 8. 飛行敵（バット）を削除する

`ENEMY_TYPES.flying`エントリ、`Enemy`クラスの飛行専用移動処理（`this.flying`分岐、上下にふわふわ揺れる動き）、`StageManager.spawnWave`の出現テーブルからの`flying`候補、`IMAGE_MANIFEST.enemySprites.flying`を削除する。

これにより項目5（飛行敵に剣を少し当てる／攻撃判定の高さ拡張）は不要になったため**取り下げる**。項目1（左右判定拡張）の`getAttackHitbox`変更は高さ（`h`）はそのまま50を維持し、幅（`w`/`x`）のみ左右対称にする。

## 9. 攻撃・能力・移動をAI自動化する（リズム判定自体は維持）

### 移動
`Player`クラスに、`isLocal`かどうかによらず「最も近い生存中の敵に向かって自動で近づく」移動AIを追加する。既存の矢印キー入力（`input.left`/`input.right`）によるプレイヤー移動は廃止する。速度は既存の`CONSTANTS.PLAYER_SPEED * this.char.speed`を流用する。

### 攻撃バーストの自動開始（スケジュール式ではなく位置ベース）
現状Zキーが担っている「バースト開始」と「ノーツ判定」のうち、「開始」を自動化する。**固定拍数ごとのスケジュールではなく**、既存の敵AIと同じ考え方（`Enemy.update`が最寄りのプレイヤーとの距離を見て攻撃範囲内に入ったら`startAttack()`する）を踏襲し、**プレイヤーキャラが最寄りの敵の攻撃間合いに入り、かつ現在バーストが動いていなければ自動で`startSwordBurst(4)`を開始する**。Zキーは`checkInput('sword')`によるノーツ判定専念になる。

### 能力バーストの自動開始
既存の`abilityCooldown`（クールダウン、現状8秒）が0になった時点で、間合いに関わらず自動で`startAbility(4)`を開始する（能力は範囲攻撃やバフが中心のため、位置条件は課さない）。Xキーは`checkInput('ability')`によるノーツ判定専念になる。

### 判定システムへの影響
`RhythmSystem`の判定ロジック（Perfect/Great/Good/Miss・コンボ・スコア）は変更しない。変わるのは「バーストを開始するタイミングの決定主体」のみで、各ノーツを正確なタイミングで叩けるかどうかは引き続きプレイヤーの入力（Z/Xキー）にかかっている。

## 今回のスコープ外（変更しない）

- ウェーブ制・BGM実音源・カウンター全廃・ノックバック演出・6キャラクター構成・UIのflexレイアウト自体は先行実装のまま維持
- 敵側の攻撃演出・耐性システムは変更しない
- LANマルチプレイの実通信は引き続き未実装
